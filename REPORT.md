## 1. Постановка сценария
Описание сценария без использования ключей идемпотентности:

1. Пользователь отправляет запрос на оплату заказа.
2. Сервер принимает запрос и выполняет его обработку, в этот момент происходит разрыв канала связи, пользователь не получает ответа от сервера.
3. Пользователь отправляет повторный запрос на оплату того же заказа, несмотря на то что он уже был обработан сервером.
4. Сервер получает запрос и далее возможно несколько сценариев: повторная оплата (критическая ошибка) или отправка ошибки. В обоих случаях пользователь не получает информации о том, что его предыдущий запрос уже был успешно обработан.

## 2. Реализация таблицы idempotency_keys
```sql
CREATE TABLE idempotency_keys (
    id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
    idempotency_key VARCHAR(255) NOT NULL,
    request_method VARCHAR(16) NOT NULL,
    request_path TEXT NOT NULL,
    request_hash TEXT NOT NULL,
    status VARCHAR(32) NOT NULL DEFAULT 'processing',
    status_code INTEGER,
    response_body JSONB,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMP NOT NULL DEFAULT NOW(),
    expires_at TIMESTAMP NOT NULL,
    CONSTRAINT idempotency_status_check CHECK (status IN ('processing', 'completed', 'failed')),
    CONSTRAINT key_endpoint_unique_check UNIQUE (idempotency_key, request_method, request_path)
);
```

Поля для идентификации запроса:
* `idempotency_key` - уникальный ключ запроса, генерируемый клиентом
* `request_method` - тип HTTP-запроса (GET, POST, ...)
* `request_path` - полный путь api-эндпойнта
* `request_hash` - хешированное тело запрос для проверки полной идентичности запросов с одинаковым idempotency_key

Данные запроса:
* `status` - статус обработки задачи (processing, completed, failed)
* `status_code` - код статуса обработки (200, 400, 500 и тд)
* `response_body` - непосредственно результат обработки, сам кеш
* `created_at`, `updated_at`, `expires_at` - метки времени 

Ограничения:
* Статус может принимать только одно из трёх значений
* Должна соблюдаться уникальность записей по совокупности трёх полей (idempotency_key, request_method, request_path) для однозначной идентификации кеша

## 3. Реализация middleware
1. Чтение `Idempotency-Key`.
2. Проверка существующей записи.
3. Обработка кейса \"тот же key + другой payload\".
4. Сохранение результата первого запроса.
5. Возврат кэшированного ответа при повторе.

## 7. Сравнение с решением из ЛР2 (FOR UPDATE)
||Идемпотентность (Idempotency Key)|Блокировка строки (FOR UPDATE)|
|-----|-------|--------|
|Цель|Защита от повторных запросов, оптимизация обработки при помощи кеша|Защита от race condition|
|Обработка повторного запроса|Возвращает тот же успешный ответ (статус 200), что и первый раз, мгновенно (из кэша).|Второй запрос будет ждать освобождения блокировки первым запросом. После этого он выполнится заново, что при отсутствии других защит приведёт к двойному списанию средств или ошибке валидации.|
|Уровень гарантии|API-слой|БД-слой|

`FOR UPDATE` защищает от конкурентного доступа внутри базы данных (если идемпотентность по какой-то причине не сработала или ключи разные), а идемпотентность улучшает пользовательский опыт (UX), возвращая мгновенный ответ при повторе сети, и снижает нагрузку на БД, предотвращая лишние транзакции, поэтому данные методы могут и должны использоваться совместно.

## 8. Выводы
1. Использование ключа идемпотентности и middleware позволяет создать дополнительный слой защиты сервиса, улучшая надёжность и комфорт в использовании. При попытке повторной оплаты пользователь не получит ошибку, а получит закешированные данные. 
2. Ключ идемпотентности генерируется на клиенте, поэтому клиент ответственен за его уникальность и неизменность данных, соответствующих ключу. 
3. Механизм идемпотентности может применяться совместно с FOR UPDATE, это делает сервис защищённым на разных уровнях и максимально улучшает удобство для пользователя.
